How to Handle and Wrap Errors in More Detail

Basics of Error Handling in Go Language
#

In the Go language, we use the error interface to handle errors, and the definition of the error interface is very simple; it only needs to include an Error() string method. However, how to pass detailed and complete error information through layers of programs, record errors, and conveniently and quickly find the root cause of errors is clearly a challenge. This article will present some personal insights on this issue. (Note: This article focuses more on expressing personal opinions. If it helps you, I recommend combining them with your own projects to build an error handling and logging system.)

type error interface {
    Error() string
}

I believe the two most important aspects of error handling are error chaining and call stacks. When an error occurs, we need to know the complete error chain at the call site and the complete call stack at the error occurrence site. With these two points, we can not only understand the interconnected error information but also directly identify the call flow and the source of the error based on the call stack. This article will not delve into the combination of both but will separately explain and demonstrate relevant mechanisms.

Error Wrapping Mechanism from Go 1.13
#

Starting from Go 1.13, the standard library provides error wrapping functionality. You can use fmt.Errorf and the %w formatting verb to wrap errors, which allows preserving the context information of the original error while adding more descriptions.

if err := do(); err != nil {
    return fmt.Errorf("failed to do..., %w", err)
}

Here is a simple demo of applying error chaining.

func main() {
    err := do()
    e := errors.Unwrap(err)
    fmt.Printf("err: %v\n", err)
    fmt.Printf("origin: %v\n", e)
}

func do() error {
    if err := learn(); err != nil {
        return fmt.Errorf("do something failed, %w", err)
    }
    return nil
}

func learn() error {
    return fmt.Errorf("can`t learn")
}

The final output is as follows:

err: do something failed, cant learn

origin: cant learn

By now, I believe you have some creative ideas. Using the error chaining mechanism not only allows passing error information but also opens up more possibilities due to the existence of Unwrap. Next, I will introduce how to define custom error types and pass additional information.

Defining Custom Error Types
#

First, we define our own structure and an interface that satisfies the methods of our structure.

type Merror interface {
    Error() string
    Extra() any
}

type merror struct {
    msg   string // Store error message
    extra int64  // Store additional information (any type)
}

// New creates a new structure
func New(msg string, extra ...int64) error {
    e := &merror{
        msg: msg,
    }
    if len(extra) > 0 {
        e.extra = extra[0]
    }
    return e
}

func (e *merror) Error() string {
    return e.msg
}

func (e *merror) Extra() any {
    return e.extra
}

// Format is used for formatted output, converting to the content we want when encountering placeholders
func (e *merror) Format(state fmt.State, verb rune) {
    switch verb {
    case 's', 'v':
        io.WriteString(state, fmt.Sprintf("msg: %s,extra: %d", e.msg, e.extra))
    }
}

Our custom merror implements the Error() string and Formatter(state, rune) interfaces. The former means that merror can be passed as an error. The latter allows us to implement our desired output format.

Example of Using Custom Error Types
#

Now let’s demonstrate the above code with a demo:

func main() {
    e := merror.New("a error", 1024) // Get a new error and pass an extra value of 1024
    fmt.Printf("%s\n", e.Error()) // Output the error message, should be "a error"

    var ex merror.Merror // Declare a variable of the custom interface to determine if the error is of type merror
    if errors.As(e, &ex) { // If it is merror, output the content returned by the Extra method and format the output of merror
        fmt.Printf("%d\n", ex.Extra())
        fmt.Printf("%v\n", ex) // Here, the %v placeholder corresponds to the case 'v' in the above Format method
    }
}

Final output:

a error 1024 msg: a error,extra: 1024

Let’s briefly summarize what we’ve covered so far:

  1. Structures that implement the Error() string method are all error.
  2. By using fmt.Errorf("%w"), errors can be encapsulated to form a call chain.
  3. Define a structure that implements the error interface and contains additional fields, implementing the fmt.Formatter interface.
  4. Use errors.As() to obtain information from a custom error.

Combining all of these, I’m sure you already have some guesses about what will happen next! Yes, we’re going to encapsulate some information in merror so that we can get the call stack information! (Thank you to the contributors of the github.com/pkg/errors library, who have allowed me to stand on the shoulders of giants.)

Obtaining and Encapsulating Call Stack Information
#

For the next steps, I won’t just show you the code and then explain it, but rather we’ll build it together step by step.

First, we need to think about how to get the call stack? The answer is runtime, also known as the runtime environment. Just hearing the name gives you a general idea of its meaning.

There is a function defined as follows:

Callers(skip int, pc []uintptr) int

Using runtime.Callers(), you can store stack information in pc. Let’s skip over the meaning of the skip parameter for now and focus on the type of pc, []uintptr. What exactly is uintptr? You might want to read this article: Deep Dive into uintptr Type

After learning the above, you can try out this function. Once you’ve tried it, let’s continue.

After learning the above, I thought it would be great to wrap it into a function to easily get the caller information, instead of defining it every time, which is quite troublesome. Alright, let’s do it!

func callers() *stack {
    const depth = 32
    var pcs [depth]uintptr
    n := runtime.Callers(0, pcs[:])
    var st stack = pcs[0:n]
    return &st
}

What? Directly passing []uintptr doesn’t look good? It’s unclear? Not easy to format? You’re right, so let’s encapsulate it further.

type stack []uintptr

func callers() *stack {
    const depth = 32
    var pcs [depth]uintptr
    n := runtime.Callers(0, pcs[:])
    var st stack = pcs[0:n]
    return &st
}

Now it looks much better. What’s next? Although we’ve added a layer, how do we convert the mysterious []uintptr into something readable and format it as a string?

Formatting and Outputting the Call Stack
#

Next, let’s bring in the runtime package again. If you’ve studied computer architecture, you should know that there is a special register used to store instruction addresses, and it increases as the program progresses—yes, it’s the pc (Program Counter), which stores the address of the next instruction to be executed. Each time an instruction is executed, the pc increments, pointing to the next address. Now, we can consider each uintptr in our stack as a pc (this requires further investigation), and using functions from the runtime package, we can format the information we obtain.

Next, let’s retrieve the file, line, and name information for a single uintptr.

func file(u uintptr) string {
    fn := runtime.FuncForPC(u)
    if fn == nil {
        return "unknown"
    }
    file, _ := fn.FileLine(u)
    return file
}

func line(u uintptr) int {
    fn := runtime.FuncForPC(u)
    if fn == nil {
        return 0
    }
    _, line := fn.FileLine(u)
    return line
}

func name(u uintptr) string {
    fn := runtime.FuncForPC(u)
    if fn == nil {
        return "unknown"
    }
    return fn.Name()
}

Now, we can extract the above information from a single uintptr. At this point, you might think it’s ready to combine with stack and prepare for formatted output, right? Sort of, but not quite. While it’s possible now, looking at the numerous functions above, is it elegant? No, it’s not! So let’s use a struct to encapsulate it.

type Frame uintptr

func (f Frame) pc() uintptr { return uintptr(f) - 1 } // Why subtract 1? Because calling a function increments the pc!

func (f Frame) file() string {
    fn := runtime.FuncForPC(f.pc())
    if fn == nil {
        return "unknown"
    }
    file, _ := fn.FileLine(f.pc())
    return file
}

func (f Frame) line() int {
    fn := runtime.FuncForPC(f.pc())
    if fn == nil {
        return 0
    }
    _, line := fn.FileLine(f.pc())
    return line
}

func (f Frame) name() string {
    fn := runtime.FuncForPC(f.pc())
    if fn == nil {
        return "unknown"
    }
    return fn.Name()
}

Now it looks much more elegant, right? Still not elegant enough? Me too. We should also implement formatting for Frame to make stack even more convenient! Let’s do it.

func (f Frame) Format(s fmt.State, verb rune) {
    switch verb {
    case 's':
        switch {
        case s.Flag('+'):
            io.WriteString(s, f.name())
            io.WriteString(s, "\n\t")
            io.WriteString(s, f.file())
        default:
            io.WriteString(s, path.Base(f.file()))
        }
    case 'd':
        io.WriteString(s, strconv.Itoa(f.line()))
    case 'n':
        io.WriteString(s, funcname(f.name()))
    case 'v':
        f.Format(s, 's')
        io.WriteString(s, ":")
        f.Format(s, 'd')
    }
}

func funcname(name string) string {
    i := strings.LastIndex(name, "/")
    name = name[i+1:]
    i = strings.Index(name, ".")
    return name[i+1:]
}

Now it’s much better. For each uintptr, we can use %+v for direct formatting. For a stack, multiple calls can be made. Below is the Format method for stack.

func (s *stack) Format(st fmt.State, verb rune) {
    switch verb {
    case 'v':
        for _, pc := range *s {
            f := Frame(pc)
            fmt.Fprintf(st, "\n%+v", f)
        }
    }
}

To make the format more uniform, let’s add some restrictions:

func (s *stack) Format(st fmt.State, verb rune) {
    switch verb {
    case 'v':
        switch {
        case st.Flag('+'):
            for _, pc := range *s {
                f := Frame(pc)
                fmt.Fprintf(st, "\n%+v", f)
            }
        }
    }
}

Integrating the Call Stack into Custom Errors
#

Alright! After all this hard work, let’s integrate stack into merror.

type merror struct {
    msg   string
    extra int64
    *stack // Note the format here, this is to allow `stack` to participate directly in the formatting of `merror`
}

func New(msg string, extra ...int64) error {
    e := &merror{
        msg:   msg,
        stack: callers(),
    }
    if len(extra) > 0 {
        e.extra = extra[0]
    }
    return e
}

Let’s test the results!

func main() {
    e := merror.New("a error", 1024)
    fmt.Printf("%s\n", e.Error())

    var ex merror.Merror
    if errors.As(e, &ex) {
        fmt.Printf("%d\n", ex.Extra())
        fmt.Printf("%+v\n", ex) // Note the change to %+v
    }
}

Output result:

a error 1024 msg: a error,extra: 1024

What’s going on? Why is there no change? Don’t panic, remember that the basis for placeholder formatting is the Format method, and the variable in our placeholder is merror.Merror and not stack. Let’s write a few more lines.

func (e *merror) Format(state fmt.State, verb rune) {
    switch verb {
    case 's', 'v':
        io.WriteString(state, fmt.Sprintf("msg: %s,extra: %d", e.msg, e.extra))
        switch {
        case state.Flag('+'):
            io.WriteString(state, fmt.Sprintf("%+v", e.stack)) // Here!
        }
    }
}

Output result:

a error 1024 msg: a error,extra: 1024 runtime.Callers /usr/local/go/src/runtime/extern.go:331 mutezebra/pkg/merror.callers /projects/mutezebra/pkg/merror/stack.go:82 mutezebra/pkg/merror.New /projects/mutezebra/pkg/merror/merror.go:22 main.main /projects/mutezebra/main.go:10 runtime.main /usr/local/go/src/runtime/proc.go:272 runtime.goexit /usr/local/go/src/runtime/asm_amd64.s:1700

Seeing this, do you feel like the job is done? Yes, that’s all I wanted to share.
